1. Introduction
Diabetes is a serious chronic condition that affects hundreds of millions of people worldwide and, if left undetected, can lead to complications ranging from cardiovascular disease to kidney failure. Because many of the warning signs show up in routine checkups, machine learning is a natural fit for building early-warning tools that can flag at-risk patients before symptoms become severe.
For this milestone I chose the Diabetes Prediction Dataset (Mustafa, 2023) from Kaggle. It combines eight demographic and clinical features from patient health records with a binary diabetes label, making it a clean and meaningful dataset for classification. The dataset is a mix of numeric and categorical variables, It also noted that dataset has class imbalance.. This also means there are genuine data-preprocessing challenges to work through, not just model selection.
This report walks through three Kaggle notebooks that use this dataset, summarises what each author did well, highlights techniques I had not yet encountered in class, and then outlines the contributions I plan to make in the mini-project.

3. Predictive Learning Task
The main goal of this project is a supervised binary classification task. In simple terms, we want to use a patient’s eight health-related features to predict whether the patient has diabetes (label = 1) or does not have diabetes (label = 0). During training, the model learns from examples where both the patient information (the feature vector) and the correct diagnosis are already known. Using these labeled examples, the algorithm learns a function that can take a new patient’s features as input and predict their likely diagnosis.
Evaluation Metrics
One thing that makes this task a bit tricky is the class imbalance. Because 91% of patients are non-diabetic, a classifier that always outputs 0 would still achieve 91% accuracy, which is obviously useless. For that reason, accuracy should always be paired with metrics that capture performance on the minority class:
MetricWhy it matters hereAccuracyOverall EstimatePrecision (class 1)How often a positive prediction is actually correctRecall (class 1)Most important clinically; missing a diabetic patient is costlyF1-ScoreBalances precision and recall into one numberROC-AUCMeasures discrimination independent of any single threshold

4.1.1 Data Preparation
This is the most thorough notebook in terms of exploratory analysis. The author organizes the EDA into three levels, moving from single variables up to interactions between multiple features:
* Univariate analysis: Histograms and count plots cover age distribution, the heavy skew toward non-smokers, and the target class imbalance.
* Bivariate analysis: Boxplots comparing BMI, age, HbA1c_level, and blood_glucose_level against the diabetes label show clear separation, particularly for the two clinical markers.
* Multivariate analysis: Pair plots and violin plots broken down by gender suggest that diabetic patients tend to be older and carry higher BMI on average.
After the EDA phase, several preprocessing decisions are made before any model is trained:
1. Duplicate removal: 3,854 exact duplicate rows are dropped.
1. Rare-value removal: The three records where gender == 'Other' are excluded, since they represent only 0.002% of the data.
2. Smoking history relabeling: The six original smoking categories are grouped into three: non-smoker (never + No Info), current, and past_smoker (ever + former + not current). This cuts down cardinality and avoids producing very sparse one-hot columns.
3. Preprocessing pipeline: StandardScaler is applied to the six numeric features, and OneHotEncoder handles gender and smoking_history, all assembled inside a ColumnTransformer.
4.1.2 Handling Class Imbalance
To tackle the 91%/9% split, the author chains SMOTE oversampling with RandomUnderSampler inside an imbalanced-learn pipeline:

from imblearn.under_sampling import RandomUnderSampler
from imblearn.pipeline import Pipeline as imbPipeline
from sklearn.ensemble import RandomForestClassifier

over  = SMOTE(sampling_strategy=0.1)          
under = RandomUnderSampler(sampling_strategy=0.5)  

clf = imbPipeline(steps=[
    ('preprocessor', preprocessor),  # ColumnTransformer: StandardScaler + OHE
    ('over',         over),
    ('under',        under),
    ('classifier',   RandomForestClassifier())
])
4.1.3 Model & Hyperparameter Tuning
A RandomForestClassifier is tuned via GridSearchCV with 5-fold cross-validation over the following grid:

5. Techniques learned
Going through these notebooks, I came across several novel methods. Here is a brief explanation of each.
5.1 SMOTE — Synthetic Minority Over-sampling Technique (Notebook 1)
Rather than duplicating existing minority-class samples, SMOTE generates new synthetic examples by interpolating between them in feature space. For each minority point , a new sample is created as:

5.2 PCA for Feature Decorrelation (Notebook 2)
PCA finds an orthogonal basis aligned with the directions of maximum variance in the feature matrix. Even when all components are retained (as in Notebook 2), the transformation still decorrelates the inputs, which can stabilise training for classifiers that are sensitive to multicollinearity. In this dataset, features like BMI and blood glucose are likely correlated since both tend to be elevated together in metabolic syndrome, so the decorrelation could be genuinely helpful.
5.3 GridSearchCV with Leak-Proof Pipelines (Notebooks 1 & 3)
GridSearchCV sweeps exhaustively over a hyperparameter grid using k-fold cross-validation. The important correctness point is that when preprocessing steps such as scaling or SMOTE are wrapped inside a Pipeline, they are fitted on each training fold independently rather than on the whole dataset. This prevents information from the validation fold leaking into the preprocessing step, which is a subtle but common mistake in public notebooks that apply the scaler before doing the train/test split.
5.4 Model Serialisation with Pickle (Notebook 2)
The author saves both trained models to disk using Python's pickle module, so they can be reloaded for inference later without retraining from scratch. Notebook 3 also imports joblib, which is generally preferred for scikit-learn models because it handles large numpy arrays more efficiently. Either approach is standard practice in any deployment-ready machine learning workflow.

6.2 Recall-Optimised Threshold Selection
In the three notebooks reviewed, the authors all use the default 0.5 decision threshold when classifying patients. However, in a medical screening context, the consequences of different types of errors are not equal. Missing a patient who actually has diabetes (a false negative) can be more harmful than incorrectly flagging someone who does not have the disease (a false positive). Because of this, it is important to adjust the decision threshold to better reflect clinical priorities.
To address this, I will first plot the Precision–Recall curve for the best-performing model. This curve shows how precision and recall change as the decision threshold is adjusted.
Next, I will identify a threshold that achieves at least 90% recall for the diabetic class. This ensures that the model correctly identifies the large majority of diabetic patients, reducing the risk of missed diagnoses.
Finally, I will report the corresponding precision at that threshold, highlighting the trade-off between capturing more true diabetic cases and increasing the number of false positives.
By selecting a threshold based on recall rather than relying on the default value, the evaluation moves beyond a single confusion matrix and instead focuses on a clinically meaningful operating point. This approach represents a contribution beyond the reviewed notebooks, which do not explore threshold tuning for medical decision-making.
7. Conclusion
When comparing the three notebooks, there are several clear patterns. One consistent finding is that HbA1c level and blood glucose level are by far the most important predictors of diabetes. In Notebook 1, these two variables alone account for 76% of the total feature importance in the Random Forest model. This aligns well with clinical practice, since these measurements are commonly used by doctors to diagnose diabetes. Other variables such as age and BMI contribute some predictive value, but their importance is noticeably smaller. Meanwhile, smoking history and gender appear to have very little influence on the predictions once the blood-based markers are included.
Another key issue across the dataset is class imbalance, with approximately 91% non-diabetic cases and only 9% diabetic cases. This imbalance can make model training more difficult and must be addressed carefully. Notebooks 1 and 2 handle this challenge in different ways. Notebook 1 uses a combination of SMOTE oversampling and undersampling, while Notebook 2 relies on XGBoost’s ability to better manage imbalanced data. Both approaches achieve high overall accuracy, typically between 95% and 97%. However, accuracy alone does not tell the whole story. The recall for the diabetic class ranges from about 0.67 to 0.80, which is an important limitation because missing diabetic patients can have serious consequences in a medical screening scenario.
Another consistent observation is the impact of hyperparameter tuning. Without tuning, models such as Decision Trees and Random Forests tend to overfit the training data, sometimes reaching nearly 100% training accuracy while performing worse on the test set. After applying GridSearchCV for hyperparameter tuning, the gap between training and validation performance becomes much smaller, indicating better generalization. Notebook 3 also highlights that the best-performing model can depend on the dataset itself. For example, on the smaller Pima dataset with 768 records, a tuned Decision Tree actually outperforms XGBoost, showing that no single algorithm consistently performs best in every situation.
Based on these observations, my mini-project will extend the existing work in several ways. I will conduct a systematic comparison of multiple models using the full 100,000-record dataset.
